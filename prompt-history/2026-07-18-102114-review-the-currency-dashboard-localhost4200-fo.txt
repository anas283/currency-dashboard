  Mobile Responsiveness Report — CurrencyDashboard (localhost:4200)

  Findings:

  [375px/390px] nav-bar (.nav-bar): overflow-x:auto with no scroll hint, 
  theme-toggle button and "updated Xs ago" text sit off-screen at x=864+ 
  (viewport only 375/390 wide) — user must guess to swipe header sideways to 
  reach them. Fix: collapse into hamburger/off-canvas menu below breakpoint 
  instead of relying on hidden inline scroll.

  [768px] nav-bar (.nav-bar): same clipping persists — toggle button (x=864–922)
  and tail of timestamp text still cut past 768px edge. Fix: extend responsive 
  nav breakpoint past 768px or redesign overflow handling, not just scroll.

  [375px/390px] page body: real horizontal scroll — document.scrollWidth 477px 
  vs 375/390px viewport (100+px overflow), caused entirely by rates-table below.
  Fix: see table finding.

  [375px] convert-card .converter__select (From/To dropdowns): 277×40px — height
  40px, under 44px tap minimum. Fix: min-height 44px.

  [375px] convert-card .btn--secondary "⇅ Swap": 99×34px — height 34px, worst 
  tap target on page, easy mis-tap. Fix: min-height 44px, adjust padding.

  [375px] rates-table sort-header__button "Base" column: 0×0 bounding box — 
  invisible but likely still focusable/tab-stop. Fix: remove from tab order or 
  restore visible hit area.

  [all widths ≤768px] rate table sort headers (Currency/Rate): 20px tall hit 
  targets — under 44px, low risk since not primary actions but still fails tap 
  guideline.

  No issues found: text truncation/overlap (none observed at any width),
  search/Base bar (always full-width, 48–50px tall, stays usable), two-column
  collapse (single-column at 375/390/768, proper two-column only at 1024 —
  correct behavior), 1024×768 has zero horizontal overflow, all nav items
  visible.

  Worst offender — rates-table (.rates-table__table):
  It's a real <table> (display:table, no overflow-x wrapper anywhere in the
  parent chain), fixed at 395px wide against a 343px content area at 375px
  viewport. The CHANGE column (.rates-table__delta-col) pins ~92px, and the name
  cell needs ~159px, so total width blows past viewport. Because there's no
  scroll container on the table itself, the entire page drags horizontally
  instead — worse, visually the row still looks like it reads fine (deceptive),
  so this ships unnoticed until someone actually measures scrollWidth or drags
  the page sideways on a real phone. Highest-risk element: fix by wrapping table
  in its own overflow-x:auto container, or better, switch to genuine
  stacked-card rows below the tablet breakpoint (screenshots show it's already
  visually trying to be card-like — just isn't structurally).
