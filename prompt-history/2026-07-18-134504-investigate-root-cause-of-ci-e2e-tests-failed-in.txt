⏺ Root cause found — race condition in ConverterComponent, not test flake.

  Failure: converter.cy.ts → converts amount test. Timeout waiting for
  [data-testid="converter-result"] to contain 211.86USD.

  Why:

  RatesService.convert() (src/app/core/services/rates.service.ts:105-121):
  async convert(amount, from, to) {
    const snap = this.snapshot();
    if (snap && ...) return amount * (rate);   // fast path, needs snapshot 
  already loaded
    ...
    return firstValueFrom(this.http.get(pair endpoint)).catch(() => null); // 
  fallback
  }

  ConverterComponent constructor (converter.component.ts:50-60):
  constructor() {
    effect(() => {
      const amount = this.amount();
      const from = this.from();
      const to = this.to();
      this.ratesService.convert(amount, from, to).then(...)
    }, { allowSignalWrites: true });
  }

  Effect deps only amount/from/to — NOT ratesService.snapshot(). Effect fires
  once at construction. If snapshot() already populated (rates loaded before
  Converter route renders) → fast path, correct value. If snapshot still null at
  that instant → falls to /pair/ endpoint, which cypress test mocks to 500,
  .catch swallows it, resultValue stays null forever. No dependency on snapshot
  means no retry when data does arrive later.

  Order between "root rates fetch resolves" and "Converter component constructs"
  is a timing race — depends on CI machine speed. Test 1 (converts amount) lost
  the race in this run, test 2 (swaps currencies) in same spec won it. Not test
  bug — 035db5d's "fix converter test race" didn't fix underlying issue, just
  changed test expectations for MYR default.

  Fix: effect must depend on this.ratesService.snapshot() too (read it inside
  effect body, even unused, so Angular tracks it), so conversion recomputes once
  rates land. Want me to patch it?

⏺ Second angle confirmed. Investigator dug into why the race is lost
  specifically on cold start: CacheService.get() (cache.service.ts:31-32) awaits
  this.ready — a schema-guard promise (line 55-61) that opens IndexedDB and may
  run clear()+set() on first use. That's slower on the very first navigation in
  a fresh browser (V8/IDB not warm yet) than on later test's revisit in the
  same Cypress-run browser instance — matching why test 1 lost the race and test
  2 in the same spec won it.

  Doesn't change the core bug, adds why the timing window exists:

  - Bug (deterministic, verified myself reading converter.component.ts:50-60):
  effect() only tracks amount/from/to signals. ratesService.snapshot() is read
  inside the .then() chain of convert(), not tracked as a signal dependency of
  the effect itself. So if snapshot() is still null when the effect fires once
  at construction, convert() falls to the mocked /pair/ 500 → null → resultValue
  never recomputes, even after snapshot arrives later.
  - Trigger (why race is winnable/losable, per investigator): CacheService's
  IndexedDB schema-guard adds variable async delay before snapshot populates —
  slower on cold/first page load than warm reload, so first spec's first test is
  most exposed.

  Fix stands: make effect depend on snapshot() too, so result recomputes once
  rates land regardless of load timing. Want me to apply that patch?
